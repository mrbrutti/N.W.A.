What the tool covers

Built for analysts who need continuity, not another one-shot scanner shell.

Inventory review

Dense host detail without losing the workspace view.

Review ports, banners, scripts, fingerprints, OS guesses, traceroute, tags, notes, and recent job history per host while keeping filters and saved views close at hand.

Diff playback

Compare checkpoints instead of eyeballing XML deltas.

N.W.A. promotes imported scans and plugin results into replayable checkpoints, then breaks changes down into added hosts, opened ports, service shifts, findings, and route movement.

Route graph

Traceroute data becomes topology you can actually scan.

Aggregate paths into a route graph, surface recurring transit nodes, and export the structure for notebooks or downstream tooling.

Background jobs

Queue follow-up work inside the same workspace.

Launch managed nmap deep enrichment or nuclei HTTP enrichment runs. Artifacts, summaries, and imported results stay tied to the workspace instead of leaking into side directories.

Exports and APIs

Ship the current state out when you need automation.

Export inventory as hosts.txt, hosts.csv, or nuclei-targets.txt. Pull machine-friendly views from /api/diff, /api/graph, /api/observations, and per-host trace JSON.

Analyst loop

Use GitHub Pages for the landing page. Run the actual workbench locally.

  1. 01

    Seed the workspace

    Start from an existing Nmap XML file or boot an empty workspace and import later.

  2. 02

    Review the map

    Filter hosts, inspect exposure, check topology, and save views for recurring slices.

  3. 03

    Enrich in place

    Queue deep nmap or nuclei jobs and let the workspace absorb the results.

  4. 04

    Replay the delta

    Compare checkpoints, annotate what changed, and export the current state for downstream use.

Quick start

go run . -f ./scan.xml -p 8080

Seeds the workspace from one XML file and hosts the UI at http://localhost:8080.

Reusable workspace

go run . -w .nwa-workspace -p 8080

Starts an empty or existing workspace directory so imports, jobs, artifacts, and analyst notes persist.

Operational notes

What GitHub Pages is doing here

This site is a static project front door: feature overview, runbook, and repo link. The Go application itself still runs locally because the live workspace needs a server, filesystem state, and optional local binaries.

Optional local tools

  • nmap for deep enrichment jobs that re-import XML into the workspace
  • nuclei for HTTP enrichment jobs and per-host findings
  • Any browser for the local UI at localhost

Useful routes

  • / dashboard and dense host review
  • /changes replayable workspace diffs
  • /graph aggregated traceroute topology
  • /scans imports, plugin queue, and recent workspace events

Machine-friendly outputs

  • /exports/hosts.csv workspace inventory
  • /exports/nuclei.txt inferred HTTP targets
  • /api/diff and /api/graph for downstream tooling
  • /api/observations and /json/<ip> for analyst context
